# Configuration

Drakkar stores runtime config in:

```txt
/data/config/settings.json
```

## Runtime layout

Persistent local data:

- `/data/config/settings.json`
- `/data/nzb-backup`

Live working paths:

- `/mnt/downloads`
- `/mnt/completed`
- `/mnt/nzb`
- `/mnt/media/movies`
- `/mnt/media/tv`

FUSE views:

- `/mnt/fuse/nzb`
- `/mnt/fuse/completed`
- `/mnt/fuse/mounted/releases`

NZB backups are not exposed through the FUSE tree.

## Public compose services

The published compose now includes:

- `frontend`
- `backend`
- `postgres`
- `valkey`
- `valkey-insight`
- `nzbhydra2`
- `apprise-api`
- `seerr`
- `bazarr`

They share one bridge network, but Drakkar still uses `settings.json` as the source of truth for integration URLs, API keys, and behavior.

## Shared API token

The Settings page exposes one shared **Drakkar API Token**.

Use it for:

- frontend to backend requests
- remote REST API calls
- GraphQL requests
- Bazarr Sonarr/Radarr-compatible requests

Remote API example:

```bash
curl -H 'x-api-token: YOUR_DRAKKAR_API_TOKEN' \
  http://HOST:8080/api/status
```

Bearer auth with the same token also works, but it is not required.

## Request sync cadence

Drakkar now treats request intake as two separate paths:

- **Seerr webhook** for immediate single approved requests
- **Periodic request sync** for broader reconciliation, duplicate cleanup, recovery, and queue seeding

Default periodic request sync interval:

```txt
15 minutes
```

That is intentionally slower than before so large libraries do not get hammered by full polling loops while the webhook already covers fast single-request delivery.
